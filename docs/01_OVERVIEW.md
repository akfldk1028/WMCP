# 01. WebMCP 프로젝트 개요

## 한 줄 요약

**WebMCP는 웹 페이지가 자기 기능을 AI 에이전트에게 "도구(Tool)"로 노출하는 브라우저 API 표준이다.**

---

## 풀어서 설명하면

### 지금의 문제

AI 에이전트(ChatGPT, Claude, Gemini 등)가 외부 서비스를 사용하려면, 그 서비스가 백엔드 서버를 별도로 만들어서 API를 제공해야 한다. 예를 들어:

```
사용자 <--채팅--> AI 에이전트 <--MCP/API--> 백엔드 MCP 서버 (Python/Node.js)
```

문제점:
- 웹 개발자는 프론트엔드(JavaScript)에 익숙한데, 별도의 백엔드 서버를 만들어야 함
- 이미 프론트엔드에 구현된 기능을 백엔드에 다시 구현해야 함 (코드 중복)
- 사용자가 보는 웹 UI와 에이전트가 조작하는 API가 따로 놈 (컨텍스트 단절)
- AI가 웹사이트를 조작하려면 스크린샷 + DOM 스냅샷 + 클릭 시뮬레이션 등 불안정한 방법을 써야 함

### WebMCP의 해결책

**웹 페이지 자체가 MCP 서버 역할을 하게 만든다.** 브라우저 안의 JavaScript로 도구를 등록하면, 브라우저의 AI 에이전트가 바로 호출할 수 있다.

```
사용자 + AI 에이전트
       |
       | (같은 브라우저 안에서)
       |
       v
   웹 페이지 (JavaScript로 도구 등록)
       |
       | navigator.modelContext.provideContext({ tools: [...] })
       |
       v
   기존 프론트엔드 비즈니스 로직 재사용
```

## 핵심 용어 정리

| 용어 | 정의 |
|------|------|
| **Agent (에이전트)** | 사용자의 목표를 이해하고 대신 행동하는 자율 비서. 보통 LLM 기반 AI 플랫폼이 구현함 |
| **Browser's Agent (브라우저 에이전트)** | 브라우저가 제공하거나 브라우저를 통해 제공되는 에이전트. 브라우저 내장, 확장 프로그램, 플러그인 등 |
| **AI Platform (AI 플랫폼)** | ChatGPT, Claude, Gemini 같은 에이전틱 비서 서비스 제공자 |
| **Tool (도구)** | 웹 페이지가 에이전트에게 노출하는 JavaScript 함수. 이름 + 설명 + 스키마 + 실행 콜백으로 구성 |
| **Backend Integration (백엔드 통합)** | AI 플랫폼이 서비스의 백엔드 서버와 직접 통신하는 기존 방식 (MCP 서버, OpenAPI 등) |
| **Actuation (액추에이션)** | 에이전트가 클릭, 스크롤, 타이핑 등 사용자 입력을 시뮬레이션하여 웹 페이지와 상호작용하는 것 |
| **Model Context Provider** | WebMCP API를 사용하여 도구를 제공하는 최상위 브라우징 컨텍스트 (=탭) |

## 목표 (Goals)

1. **Human-in-the-loop 워크플로우**: 사용자가 AI에게 작업을 위임하면서도 웹 페이지에 대한 가시성과 제어권을 유지
2. **AI 에이전트 통합 단순화**: UI 액추에이션 대신 잘 정의된 JavaScript 도구로 더 안정적으로
3. **개발자 부담 최소화**: 기존 JavaScript 코드를 최소한으로 수정해서 도구로 노출
4. **접근성 개선**: 보조 기술이 기존 접근성 트리 너머의 기능에 접근할 수 있게

## 비목표 (Non-Goals)

1. **헤드리스 브라우징**: 사람 없이 백그라운드에서 브라우저 실행하는 시나리오 (미래 과제)
2. **완전 자율 에이전트**: 사람 감독 없이 동작하는 에이전트는 A2A 프로토콜의 영역
3. **백엔드 통합 대체**: 기존 MCP를 대체하는 게 아니라 보완
4. **인간 인터페이스 대체**: 에이전트 도구는 사용자 인터랙션을 증강하는 것이지 대체가 아님

## WebMCP가 해결하는 진짜 문제

### 기존 방식의 한계

```
[에이전트가 웹사이트를 조작하는 기존 방법들]

1. 스크린샷 분석 -> 느리고 부정확
2. DOM 스냅샷 읽기 -> 구조 변경에 취약
3. 접근성 트리 활용 -> 많은 사이트가 미구현
4. 클릭/타이핑 시뮬레이션 -> 불안정, 다단계 작업 느림
5. 별도 MCP 서버 구축 -> 개발 부담, 코드 중복
```

### WebMCP로 달라지는 것

```
[WebMCP 이후]

1. 웹 개발자가 JavaScript로 도구 정의 -> 친숙한 언어
2. 기존 프론트엔드 함수 재사용 -> addStamp() 같은 함수를 그대로
3. 사용자와 에이전트가 같은 UI 공유 -> 컨텍스트 일관성
4. 브라우저가 권한 중재 -> 세밀한 권한 제어
5. 상태 변화가 UI에 즉시 반영 -> 사용자가 에이전트 작업을 실시간 확인
```

## 핵심 설계 원칙

1. **코드 재사용**: 기존 프론트엔드 JavaScript를 최대한 활용
2. **공유 컨텍스트**: 사용자, 에이전트, 웹 페이지가 같은 UI와 상태를 공유
3. **점진적 향상**: WebMCP는 기존 인간 인터페이스를 증강하지 대체하지 않음
4. **브라우저 중재**: 브라우저가 신뢰 경계를 관리하고 권한을 중재
5. **MCP 정렬**: MCP 프리미티브와 호환되지만 웹 플랫폼에 맞게 조정
